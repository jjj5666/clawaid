# OpenClaw Doctor - 架构设计

## 核心理念

像顶级工程师一样诊断：观察 → 假设 → 尝试 → 根据新信息收敛 → 修复。
不预设问题清单，核心是AI诊断能力。

## 用户体验

- `npx openclaw-doctor` → 自动打开本地网页
- 全程用户不需要输入任何东西（除非需要 API key 等信息）
- Google 风，干净可信
- 像跟聪明工程师朋友聊天：每一步告诉你在想什么

## 信任设计（API Key 等敏感信息）

如果诊断发现需要用户提供 API key 等敏感信息：
- 页面上明确说明：所有数据只在你本地处理，不发送到任何外部服务器
- API key 用于调用 AI 模型进行诊断分析
- 输入框旁边显示"🔒 仅本地使用，不存储"
- 代码完全开源，用户可以自行验证
- 如果用户的 openclaw 里已配置好模型，直接复用，不需要额外输入

## 三层循环架构

```
┌─ 大循环（方向）──────────────────────────────────┐
│                                                   │
│  首次：自动采集 → AI 分析 → 展示假设 → 一键修复   │
│                                                   │
│  ┌─ 中循环（执行）×3 ────────────────────┐       │
│  │                                        │       │
│  │  观察 → 推理 → 动作 → 检查              │       │
│  │  用户实时看到进度                        │       │
│  │  自动进行，不打断用户                    │       │
│  │                                        │       │
│  │  修好了？→ ✅ 结束                      │       │
│  │  3轮没修好？→ 退出中循环 ↓              │       │
│  └────────────────────────────────────────┘       │
│                                                   │
│  META-THINK（元思考）                              │
│  "我的方向对吗？之前3轮数据说明了什么？             │
│   根因判断是不是错了？换方向。"                     │
│                                                   │
│  带着新方向 → 重新进入中循环                        │
│                                                   │
│  3次元思考（9轮）都没解决？→ 问用户                 │
└───────────────────────────────────────────────────┘
```

## 五个阶段

### 1. OBSERVE（采集 - 只读，静默）
全自动，零风险：
- `openclaw status`
- `openclaw gateway status`
- 找最全的日志，智能读取
- `openclaw config get` 当前配置
- 端口占用检查
- 进程检查（重复 gateway）
- plist 环境变量（代理残留）
- Node 版本、OpenClaw 版本
- 系统信息

### 2. THINK（AI 分析 - opus-or）
把采集数据喂给 opus-or，强制思考链：
1. 观察到什么异常？（事实）
2. 指向什么根因？（不是症状）
3. 有没有可能判断错了？其他可能？
4. 最小修复方案？
5. 风险评估？会不会弄坏别的？
6. 能不能用官方 CLI 完成？

### 3. SHOW（展示给用户）
用人话展示诊断结果和修复计划。一个按钮"一键修复"。

### 4. ACT（执行修复）
优先级：
1. 🥇 官方 CLI（openclaw gateway restart、openclaw config set 等）
2. 🥈 系统命令（kill、lsof 等）
3. 🥉 改文件（最后手段，必须先备份）

执行时用户实时看到进度。

### 5. VERIFY（验证）
重跑检查，确认问题消失。

## 前后端通信

- 后端通过 SSE (Server-Sent Events) 实时推送进度到前端
- 前端只有一个页面，根据状态切换显示
- 需要用户输入时（如 API key），推送一个 input_required 事件

## 页面状态

1. **诊断中** - 进度条+正在检查的项目
2. **发现问题** - 人话描述问题 + 一键修复按钮
3. **修复中** - 实时进度，每步都有解释
4. **修好了** - 总结 + "请我喝杯咖啡" 链接
5. **没修好** - 诚实说明 + 复制诊断报告 + Discord 链接
6. **需要输入** - 简洁输入框 + 信任标识

## AI 调用

- 模型：优先使用 openclaw 已配置的 API key/模型
- 如果 openclaw 配置不可用，要求用户输入 OpenRouter API key
- 调用方式：直接 HTTP 调 OpenRouter API（不依赖 openclaw gateway，因为 gateway 可能就是坏的）
- 模型选择：opus-or 做诊断（需要强推理能力）

## 关键约束

1. 每轮自动执行多个动作，用户看着就行
2. 第一次展示假设后一键开始，后续不打断
3. 3轮没好 → 元思考换方向 → 再3轮
4. 9轮没好 → 才问用户
5. 优先 CLI，最后才改文件
6. 改文件前必须备份
7. 每次失败都是信息 → 排除假设 → 缩小范围
8. 不预定义问题清单 → 靠 AI 诊断能力
9. 没修好 → 诚实说 + 给诊断报告 → 不收钱
